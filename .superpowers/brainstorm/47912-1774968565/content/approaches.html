架构方案选择

核心问题：Swift HUD 如何从 TypeScript 状态管理库获取数据？

A

文件 IPC（推荐）

TypeScript 库将状态写入 ~/.token-hud/state.json，Swift HUD 用 FSEvents 监听文件变化，零延迟响应。

优点

  • 无需后台守护进程
  • 实现最简单
  • 离线也能工作
  • 任何语言都能写入/读取

缺点

  • 高频更新时有 I/O 开销
  • 需约定 JSON schema
TS 库 → 写入 state.json → FSEvents → Swift HUD 重绘
B

本地 HTTP 守护进程

TypeScript 库包含一个 daemon 模式，在本地启动 HTTP 服务器（如 localhost:7654），Swift HUD 定期轮询获取数据。

优点

  • 支持实时推送（WebSocket）
  • 可支持多个消费者
  • 接口更灵活（REST API）

缺点

  • 需要管理 Node.js 守护进程
  • 用户需要 node 环境
  • 进程崩溃 HUD 就断线
TS 守护进程 → HTTP/WS → Swift HUD 轮询/订阅
C

文件 IPC + HTTP 混合

持久配额用文件存储，当前 session 实时数据用轻量 HTTP（仅当有活跃会话时才启动 server）。

优点

  • 持久数据离线可用
  • 活跃 session 时实时更新
  • 最接近真实使用场景

缺点

  • 两套机制，稍微复杂
  • 需要区分"有 session"和"无 session"状态
配额: state.json → FSEvents → HUD
Session: HTTP/WS → 实时更新 HUD
